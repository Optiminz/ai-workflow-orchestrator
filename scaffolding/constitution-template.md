# [PLACEHOLDER — e.g., Sales Workflow / Editorial / Research] Constitution

**Purpose:** Define non-negotiable standards that govern every output this workflow produces. These rules exist because violating them produces work that fails in the real world — not just work that scores poorly.

[PLACEHOLDER — e.g., Generic content that could apply to any prospect in any industry. No specific signals used. Could have been written without any research.]

[PLACEHOLDER — e.g., Uses some prospect signals but misses the most important ones. Industry is correct but role, timing, or pain point is wrong.]

## Section 5: Evolution Rules

### When the Constitution Updates

The constitution evolves based on evidence, not opinion. These are the only valid triggers:

1. **Score <50 on any project** — automatic α-phase trigger. Evolutionist analyses root cause and proposes changes.
2. **Same dimension scores <10 on three consecutive projects** — flag for review. May indicate a systemic problem not visible in total scores.
3. **Quarterly review** — human-initiated review of Kill List, scoring rubrics, and phase constraints.

### What Can Change

- Kill List additions (based on observed violations across projects)
- Scoring rubric language (clarify descriptions based on edge cases encountered)
- Phase gate requirements (add requirements based on failures; simplify based on evidence they're unnecessary)
- Domain-specific standards (update based on what the market responds to)

### What Cannot Change

- **The five-dimension scoring framework** — changing dimensions invalidates historical comparisons
- **The APPROVE/REVISE/RESTRUCTURE/RELEASE threshold structure** — these thresholds are calibrated across projects; arbitrary changes break the system
- **The memory protocol** — the four-tier memory system is a core architectural decision
- **Human approval requirement for final output** — AI never ships without human confirmation

### How Changes Are Made

1. Evolutionist (α-phase) proposes specific changes with evidence (the project that failed, the pattern observed)
2. Human reviews the proposal
3. Human approves, modifies, or rejects
4. Approved changes are applied to this file
5. Version number is incremented
6. Entry is added to the Version History below
